  发现的 Bug

  1. SessionServiceImpl.java — 批量复制模板遗漏 UUID 赋值 (严重)

  SessionServiceImpl.java:369 — batchCopyFromTemplates 中验证了 item 的 uuid，但从未将其赋值给 newSession。复制出来的届次永远不会有通知文件 UUID。

  // 第358行：此时 newSession.getUuid() 永远是 null，所以这个 if 永远不会执行
  if (StringUtils.isNotBlank(newSession.getUuid())) {  // BUG: 从未设置过uuid
      validateAndFinalizeNoticeUuid(newSession.getUuid());
  }

  2. CompetitionApplyServiceImpl.java — 驳回时后端覆盖为 null

  审核页面的 handleReject 只传了 {status: "2", auditRemark: "..."}，但 reviewCompetitionApply 方法会覆盖 name、year、session 等字段为 null（第765-768行），可能导致申请记录数据丢失。

  3. CompetitionMapper.xml — 多选查询最多支持4个值

  CompetitionMapper.xml:86 中 UNION ALL SELECT 1 UNION ALL SELECT 2 UNION ALL SELECT 3 UNION ALL SELECT 4 硬编码了4个数字表。如果赛事类别/级别/标签超过4个逗号分隔的值，SQL 查询将静默失败。

  4. CompetitionController.java — ClassCastException 风险

  CompetitionController.java:156 将 params 中的 sessionIds 强转为 List<Integer>。如果前端发送 Long 类型值会抛出 ClassCastException。

  5. handleDeleteCompetitionDeptRel 索引不稳定

  competition/index.vue:440 用动态计算的 row.index 做删除标记，增删行后 index 会变化，可能导致删错行。

  6. normalizeAttachments 中 "0" 作为 UUID 兜底值

  CompetitionApplyServiceImpl.java:263 — String uuid = ... item.getUuid() : "0" — 用字符串 "0" 作为兜底UUID，会被当作有效值写入数据库，产生脏数据。

  代码质量问题

  7. 大量代码重复 — 申请页和审核页约60%逻辑相同

  competitionapply/index.vue 和 competitionapply-review/index.vue 共享附件管理、PDF预览、文件下载等逻辑，完全重复。应抽取为 composable 或共享组件。

  8. N+1 查询问题

  CompetitionMapper.xml:28 用 <collection select="selectCompetitionDeptRelList"> 为每行赛事单独查询部门关系，列表分页时产生 N+1 次查询。

  9. 无后端参数校验

  Controller 方法没有 @Valid 注解，完全依赖前端校验。恶意请求可直接发送空数据。

  10. Domain 对象职责过重

  Competition.java 和 CompetitionApply.java 同时承担：JPA实体映射、Excel导出注解、JSON序列化、查询参数接收、表单数据接收，违反单一职责原则。

  11. 前端 watch 触发过多 API 调用

  competition-apply/index.vue:1016 中 watch([name, year, session]) 在每次输入变化时都触发 checkSessionStatus 和 checkDuplicateApplication，缺乏防抖处理。

二、修复方案

  Bug 1: UUID 未赋值 → 在 newSession 构造后添加：
  newSession.setUuid(item.getUuid()); // 补充这一行

  Bug 2: 驳回覆盖 null → 方案A（推荐）：后端只在 status="1" 时才更新 name/year 等业务字段，驳回时只更新 status + auditRemark。

  Bug 3: 硬编码 UNION → 改为动态生成数字表，或直接用 FIND_IN_SET 方案：
  <foreach collection="categoryList" item="cat" separator=" AND ">
      FIND_IN_SET(#{cat}, c.category)
  </foreach>

  Bug 4: Integer 强转 → 改为 Number 类型后转 Long：
  List<Long> sessionIds = ((List<?>) params.get("sessionIds")).stream()
      .map(o -> ((Number) o).longValue()).collect(Collectors.toList());

  三、代码质量问题

  1. 60% 重复代码 — 申请/审核两个页面

  两者共享：
  - 附件管理逻辑 (Tab、预览、下载、删除)
  - PDF 预览 (fetchPdfObjectUrl, loadAttachmentPreview)
  - 文件操作 (handleOpenDetail, handleDownloadAttachment)
  - 附件数据填充 (fillAttachmentData, buildAttachmentList)

  建议抽取为 useCompetitionAttachments composable。

  2. N+1 查询

  CompetitionMapper.xml 用 <collection select="..."> 为每行单独查部门关系。10条数据 = 11次查询。建议改为 JOIN + 手动组装，或者用 MyBatis 的 association 替代 select。

  3. 前端防抖缺失

  watch([name, year, session]) 在每次按键时触发2个API。建议加 500ms debounce。

  4. 无后端校验

  建议至少在 Controller 层用 @Validated + @NotNull/@NotBlank 做基础校验。
