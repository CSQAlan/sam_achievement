## 一、严重漏洞 (CRITICAL)

### 1.1 硬编码数据库凭据

**文件:** `ruoyi-admin/src/main/resources/application-druid.yml` (第9-12行)

```yaml
url: jdbc:mysql://121.36.231.106:3306/ruoyi?useUnicode=true&...
username: ruoyi1
password: Ruoyi@123456
```

- **严重性:** 严重
- **类别:** 敏感信息泄露
- **描述:** 数据库用户名和密码以明文形式硬编码在配置文件中，代码提交到Git仓库后凭据永久暴露。数据库地址 `121.36.231.106` 为公网IP，任何人获取此凭据均可直接连接生产数据库。
- **利用场景:** 攻击者通过源码泄露、Git仓库访问或反编译jar包即可获取数据库凭据，直接远程连接数据库执行任意SQL操作。
- **修复建议:** 
  1. 立即修改数据库密码
  2. 将敏感凭据移至环境变量或外部配置中心（如Spring Cloud Config、Vault）
  3. 使用 `${DB_PASSWORD}` 占位符，由部署环境注入
  4. 将 `application-druid.yml` 加入 `.gitignore`

### 1.2 弱JWT签名密钥

**文件:** `ruoyi-admin/src/main/resources/application.yml` (第118行)

```yaml
token:
  secret: abcdefghijklmnopqrstuvwxyz
```

- **严重性:** 严重
- **类别:** 认证绕过
- **描述:** JWT签名密钥为 `abcdefghijklmnopqrstuvwxyz`（26个小写字母），熵值极低。HS512算法要求至少64字节密钥。攻击者可轻易暴力破解此密钥并伪造任意用户的JWT令牌。
- **利用场景:** 
  1. 攻击者获取一个合法JWT令牌后，破解签名密钥
  2. 伪造任意用户（包括管理员）的JWT令牌
  3. 猜测或枚举Redis中的session UUID，劫持活跃会话
- **修复建议:** 
  1. 生成至少64字节的随机密钥：`openssl rand -base64 64`
  2. 通过环境变量注入，不要在配置文件中明文存储
  3. 考虑使用RSA/ECDSA非对称签名替代HS512

### 1.3 路径遍历漏洞 - 任意文件读取

**文件:** `ruoyi-admin/src/main/java/com/ruoyi/achievement/controller/AttachmentController.java` (第42-66行)

```java
@GetMapping("/download/notice")
public void downloadNotice(String fileName, HttpServletResponse response) {
    String downloadPath = RuoYiConfig.getProfile()
        + StringUtils.substringAfter(fileName, Constants.RESOURCE_PREFIX);
    // 没有任何路径遍历检查!
    FileUtils.writeBytes(downloadPath, response.getOutputStream());
}
```

- **严重性:** 严重
- **类别:** 路径遍历 / 任意文件读取
- **描述:** `fileName` 参数直接来自用户请求，未对 `../` 序列做任何过滤。`StringUtils.substringAfter()` 仅去掉 `/profile` 前缀，保留路径遍历序列。标准下载接口 `CommonController.download` 有 `checkAllowDownload()` 防护，但此接口完全缺失。
- **利用场景:**
  ```
  GET /attachment/download/notice?fileName=/profile/../../../../etc/passwd
  GET /attachment/download/notice?fileName=/profile/../../../../windows/win.ini
  ```
- **修复建议:** 
  1. 添加 `checkAllowDownload()` 调用过滤 `..` 路径
  2. 对 `fileName` 进行标准化（`FilenameUtils.normalize()`）后再拼接
  3. 验证最终路径必须在允许的目录范围内

### 1.4 FastJSON2 AutoType反序列化风险

**文件:** `ruoyi-framework/src/main/java/com/ruoyi/framework/config/FastJson2JsonRedisSerializer.java` (第21,38,50行)

```java
static final Filter AUTO_TYPE_FILTER = JSONReader.autoTypeFilter(Constants.JSON_WHITELIST_STR);
// 序列化时写入 @type 类名
JSON.toJSONString(t, JSONWriter.Feature.WriteClassName)
// 反序列化时使用白名单过滤器
JSON.parseObject(str, clazz, AUTO_TYPE_FILTER);
```

**白名单配置:** `ruoyi-common/src/main/java/com/ruoyi/common/constant/Constants.java` (第161行)
```java
public static final String[] JSON_WHITELIST_STR = { "com.ruoyi" };
```

- **严重性:** 严重
- **类别:** 反序列化远程代码执行
- **描述:** Redis序列化使用FastJSON2的AutoType机制，白名单 `com.ruoyi` 覆盖了整个项目包（包括所有自定义业务类、控制器、服务、配置类等）。序列化数据中写入 `@type` 字段，反序列化时自动实例化指定类。如果攻击者能向Redis写入数据（通过SSRF、Redis未授权访问等），可构造恶意 `com.ruoyi.*` 类触发反序列化攻击链。
- **利用场景:** 
  1. Redis未设置密码（`application.yml` password为空），若Redis端口暴露可被直接写入
  2. 通过SSRF漏洞向Redis注入恶意序列化数据
  3. 构造特殊 `com.ruoyi.*` 类触发getter/setter中的危险操作
- **修复建议:** 
  1. 为Redis设置强密码
  2. 收窄AutoType白名单至最小必要类
  3. 考虑使用 `JSONB` 二进制格式替代 `@type` 文本格式
  4. Redis绑定127.0.0.1或内网IP，不暴露到公网

### 1.5 GenTableMapper SQL注入

**文件:** `ruoyi-generator/src/main/resources/mapper/generator/GenTableMapper.xml` (第175行)

```xml
<update id="createTable">${sql}</update>
```

**控制器入口:** `ruoyi-generator/src/main/java/com/ruoyi/generator/controller/GenController.java` (第130行)

```java
public AjaxResult createTableSave(String sql) {
    SqlUtil.filterKeyword(sql);  // 黑名单，可绕过
    // ...
}
```

- **严重性:** 严重
- **类别:** SQL注入
- **描述:** 代码生成器的建表功能直接接收用户输入的SQL字符串，使用 `${sql}` 拼接（而非 `#{sql}` 参数化）。虽有 `SqlUtil.filterKeyword()` 黑名单过滤和 `MySqlCreateTableStatement` 类型检查，但黑名单可被多种方式绕过（注释插入、大小写变换、编码绕过）。配合 `multi-statement-allow: true` 的Druid配置，攻击者可在CREATE TABLE后追加任意SQL语句。
- **利用场景:** 具有管理员权限的攻击者（接口有 `@PreAuthorize("hasRole('admin')")` 保护）可执行 `; DROP TABLE xxx; --` 等语句。若管理员账户被攻破或内部人员恶意操作，可导致数据丢失。
- **修复建议:** 
  1. 使用 `#{sql}` 替代 `${sql}`（参数化绑定）
  2. 使用SQL白名单验证替代黑名单过滤
  3. 将 `multi-statement-allow` 设为 `false`
  4. 限制管理员只能通过数据库管理工具执行DDL

## 二、高危漏洞 (HIGH)

### 2.1 CORS跨域配置允许所有来源

**文件:** `ruoyi-framework/src/main/java/com/ruoyi/framework/config/ResourcesConfig.java` (第56-65行)

```java
config.addAllowedOriginPattern("*");  // 允许所有来源
config.addAllowedHeader("*");          // 允许所有请求头
config.addAllowedMethod("*");          // 允许所有HTTP方法
config.setAllowCredentials(true);
```

- **严重性:** 高
- **类别:** CSRF / 跨域安全
- **描述:** CORS配置允许所有来源、所有方法和所有请求头，同时 `allowCredentials=true`。这完全绕过了浏览器的同源策略保护。任何恶意网站均可向该后端发送跨域请求，结合JWT Cookie非HttpOnly的配置，可实现CSRF令牌窃取。
- **利用场景:** 用户访问恶意网站时，该网站的JavaScript可向应用后端发送请求，利用用户浏览器中的认证Cookie执行未授权操作。
- **修复建议:** 
  1. 将 `allowedOriginPatterns` 限制为前端部署的具体域名
  2. 如果使用 `allowCredentials(true)`，则不能使用 `*` 通配符
  3. 限制 `allowedMethods` 为实际需要的方法（GET/POST/PUT/DELETE）
  4. 限制 `allowedHeaders` 为实际需要的请求头

### 2.2 自定义模块完全不在XSS过滤器保护范围内

**文件:** `ruoyi-admin/src/main/resources/application.yml` (第152-158行)  
**文件:** `ruoyi-common/src/main/java/com/ruoyi/common/filter/XssFilter.java` (第63-64行)

```yaml
xss:
  enabled: true
  excludes: /system/notice
  urlPatterns: /system/*,/monitor/*,/tool/*  # 仅覆盖框架模块!
```

```java
// GET和DELETE请求完全跳过XSS过滤
if (method == null || HttpMethod.GET.matches(method) || HttpMethod.DELETE.matches(method)) {
    return true;
}
```

- **严重性:** 高
- **类别:** XSS (跨站脚本攻击)
- **描述:** XSS过滤器的 `urlPatterns` 仅覆盖 `/system/*`、`/monitor/*`、`/tool/*`，**不包括**自定义业务模块的URL路径：

| 模块 | URL前缀 | XSS保护 |
|------|---------|---------|
| 成就管理 | `/achievement/**` | ❌ 无保护 |
| 学生管理 | `/student/student/**` | ❌ 无保护 |
| 教师管理 | `/achievement/teacher/**` | ❌ 无保护 |
| 附件上传 | `/attachment/**` | ❌ 无保护 |
| 竞赛管理 | `/competition/competition/**` | ❌ 无保护 |
| 竞赛申请 | `/competition-apply/**` | ❌ 无保护 |
| 竞赛届次 | `/session/session/**` | ❌ 无保护 |
| 报销查询 | `/reimbursement/achievement/**` | ❌ 无保护 |

同时GET请求完全跳过XSS过滤，增加了通过查询参数注入的攻击面。

- **利用场景:** 攻击者通过自定义模块POST接口提交包含HTML/JavaScript的数据（如成果名称、竞赛名称、审核意见等），数据直接存入数据库，在管理页面展示时触发存储型XSS。
- **修复建议:** 
  1. 将URL pattern扩展为 `/*` 覆盖所有接口，或用排除列表替代
  2. 修复GET/DELETE跳过过滤的逻辑
  3. 为所有自定义领域对象的字符串字段添加 `@Xss` 注解
  4. 在控制器中使用 `@Validated` 进行参数验证

### 2.3 自定义控制器全部缺少输入验证

**文件:** `ruoyi-admin/src/main/java/com/ruoyi/achievement/controller/SamAchievementController.java`
`ruoyi-admin/src/main/java/com/ruoyi/competition/controller/CompetitionController.java`
`ruoyi-admin/src/main/java/com/ruoyi/competitionapply/controller/CompetitionApplyController.java`
以及所有reimbursement包下的控制器

- **严重性:** 高
- **类别:** 输入验证缺失
- **描述:** 所有自定义业务控制器（成就、竞赛、申请、报销）的 `add()`、`edit()` 方法均未使用 `@Validated` 或 `@Valid` 注解，也未使用 `BindingResult` 处理验证结果。`@RequestBody` 参数直接接受数据，没有经过任何验证。

```java
// 所有模块都是这样 - 无验证
@PostMapping
public AjaxResult add(@RequestBody SamAchievement samAchievement) { ... }
@PutMapping
public AjaxResult edit(@RequestBody Competition competition) { ... }
```

- **利用场景:** 攻击者可以提交超长字符串、特殊字符、非法枚举值等数据，可能导致数据库截断错误、存储型XSS、业务逻辑错误等。
- **修复建议:** 
  1. 为所有 `@RequestBody` 参数添加 `@Validated` 注解
  2. 在领域对象中添加Bean Validation注解（`@NotBlank`、`@Size`、`@Pattern`等）
  3. 为字符串字段添加 `@Xss` 注解防止脚本注入

### 2.6 存储型XSS - HTML文件上传+公开访问

**文件:** `ruoyi-common/src/main/java/com/ruoyi/common/utils/file/MimeTypeUtils.java` (第29-39行)
**文件:** `ruoyi-framework/src/main/java/com/ruoyi/framework/config/SecurityConfig.java` (第120行)

```java
// MimeTypeUtils: HTML扩展名在允许列表中
public static final String[] DEFAULT_ALLOWED_EXTENSION = {

```

- **严重性:** 高

- **描述:** 文件上传允许HTML/HTM格式，且上传目录通过 `/profile/**` 公开映射可无需认证访问。攻击者上传含有恶意JavaScript的HTML文件后，任何人访问该URL即触发XSS，可能窃取同源下的Cookie或LocalStorage数据。

  2. 为上传文件添加 `Content-Disposition: attachment` 头，强制下载而非浏览器解析
  3. 为 `/profile/**` 路径添加认证要求

### 2.7 CAS认证Cookie安全问题

**文件:** `ruoyi-admin/src/main/java/com/ruoyi/demo/DemoController.java` (第121-123行)

```java
Cookie cookie = new Cookie("Admin-Token", token);
cookie.setHttpOnly(false);    // JavaScript可读取 → XSS可窃取token
// cookie.setSecure(true);   // 被注释: "本地开发先不加 secure"
cookie.setPath("/");
```

- **严重性:** 高
- **类别:** 会话劫持
- **描述:** CAS登录后的JWT令牌存储在Cookie中，`HttpOnly=false` 使JavaScript可以读取，且 `Secure` 未设置。结合2.6中的存储型XSS漏洞，攻击者可直接窃取JWT令牌，无需破解签名密钥即可以受害者身份操作。
- **利用场景:** 
  1. 攻击者通过XSS执行 `document.cookie` 获取 `Admin-Token`
  2. 将token发送至攻击者服务器
  3. 攻击者使用token伪造身份进行操作
- **修复建议:** 
  1. 设置 `cookie.setHttpOnly(true)`
  2. 设置 `cookie.setSecure(true)`（HTTPS部署时）
  3. 设置 `cookie.setSameSite("Strict")` 防止CSRF
  4. 如果前端需要读取token，考虑存储在 `Authorization` 请求头而非Cookie

### 2.8 CAS自动注册无审批机制

**文件:** `ruoyi-admin/src/main/java/com/ruoyi/demo/DemoController.java` (第99-109行)  
**文件:** `ruoyi-admin/src/main/java/com/ruoyi/demo/service/impl/CasUserProvisioningService.java` (第60-69行)

```java
// 任何CAS用户首次登录即自动创建账户
if (user == null) {
    user = casUserProvisioningService.provisionOrUpdate(loginName, attributes);
}

// 角色分配基于学号长度
if (id.length() == 8) {
    roleId = 103L; // 教师
} else if (id.length() == 13) {
    roleId = 3L;   // 学生
}
```

- **严重性:** 高
- **类别:** 权限提升
- **描述:** 任何拥有有效学校CAS账户的人首次登录即自动在系统中创建账户。角色分配基于学号/工号长度（8位=教师，13位=学生），这种判断逻辑可能被利用：若存在学号为8位的学生或可预测的工号，可能导致普通学生获得教师权限。
- **利用场景:** 
  1. 任何学校CAS用户均可无条件创建系统账户
  2. 若CAS系统中存在长度为8的学生账号（如遗留数据），会自动获得教师角色
  3. 若有人可控制CAS返回的属性，可修改学号获得更高权限

### 2.11 Excel文件上传无内容类型验证
